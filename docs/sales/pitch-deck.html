04 / 18
Bangladesh-first compliance

Mushak 6.3 + 6.8.
Atomic. Sealed. Audit-ready.

Every paid order auto-issues a Mushak-6.3 invoice with an atomic, sequential, never-reused, never-skipped serial. Every refund auto-issues a Mushak-6.8 credit / debit note tied to the original. Sequence allocation is a single SQL INSERT … ON CONFLICT DO UPDATE … RETURNING — concurrent terminals can't race for the same number, even on a Friday-night rush.

Every issued invoice freezes a JSON snapshot at the moment of issue. A reprint three months later is legally identical even if your menu prices or VAT rate changed in the meantime.

  • BIN validation at branch setup
  • Per-fiscal-year segregation
  • Per-branch sequence pools
  • Mushak Register report (NBR-ready printout)
  • Buyer + seller blocks auto-populated
  • VAT-rate snapshot on every line
  • Reprint stays legal forever
05 / 18
Offline-first desktop

Works when the WiFi
doesn't.

The Windows cashier app is a native Electron build with a SQLite outbox underneath every mutation. When the internet drops mid-rush, the cashier types the order, the kitchen ticket prints to the thermal printer, the cash drawer kicks, the receipt prints, and the Mushak invoice number is reserved locally — all without the server.

When connectivity returns the outbox drains in FIFO order with idempotent retry. Synthetic ids rebind to real server ids. Already-paid orders return the exact same response from the server's idempotency cache. No double-charge. No double-Mushak.

  • Local SQLite outbox per terminal
  • Synthetic order ids bound to real ids on drain
  • Idempotency-Key on every mutation
  • Silent ESC/POS thermal printing (no driver dialog)
  • Cash-drawer kick on cash payment
  • Token refresh + retry on drain
  • DPAPI-encrypted device credentials
  • Per-terminal pairing + remote revoke
